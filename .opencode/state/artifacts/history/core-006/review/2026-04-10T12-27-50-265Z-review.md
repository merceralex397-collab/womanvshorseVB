### AC-2: Each variant extends HorseBase with distinct stat overrides or behavior
**Status**: PASS

**Notes**: Both variants share `horse_base.gd` (script = ExtResource("res://scenes/enemies/horse_base.gd")) with `class_name HorseBase`. This is valid GDScript — multiple scenes can reference the same script file.

Stat overrides are distinct and significantly differ from HorseBase defaults (health=50, speed=100.0, contact_damage=10):

| Variant | health | speed | contact_damage |
|---------|--------|-------|----------------|
| HorseBase | 50 | 100.0 | 10 |
| HorseFast | 30 | 180.0 | 8 |
| HorseTank | 120 | 60.0 | 20 |

Exported variables on the root node (`health`, `speed`, `contact_damage`) can be preset per-instance in the scene file — this is standard Godot 4.6 behavior and matches how `horse_base.tscn` already works.

The plan correctly notes (line 31): *"No new GDScript files needed — variants use horse_base.gd directly with preset exported values."* This is valid for stat-only variants. If custom behavior (e.g., charge attack) is needed later, a dedicated `horse_fast.gd` / `horse_tank.gd` extending HorseBase can be created.

**Status**: PASS

**Notes**: Two independent visual distinction vectors are used:

This is acceptable and explicitly noted in the plan (line 17: *"modulate with dark brown tint Color(0.8, 0.6, 0.4) for extra distinction"*). The scale difference (0.8 vs 1.3) is sufficient to distinguish variants even if sprite is same.

## Integration Assessment

### Architecture
Variant scenes duplicate `horse_base.tscn` structure (CharacterBody2D + AnimatedSprite2D + CollisionShape2D + HurtboxArea) and reference the same `horse_base.gd` script with preset exported values. This is a valid Godot 4.6 pattern.

### WaveSpawner Changes
The plan correctly identifies minimal invasive changes:
1. Rename `enemy_scene` → `base_enemy_scene` (backward compatible)
2. Add `enemy_variants: Array[PackedScene]` export
3. Add `variant_intro_wave: int = 3` export
4. Update `_spawn_wave()` with conditional selection logic

The stat override mechanism is preserved: `enemy.health = health; enemy.speed = speed; enemy.contact_damage = damage` works on all HorseBase instances regardless of which packed scene was used.

### Per-variant wave scaling
The plan mentions (lines 61–63) that Fast Horse gets +10% speed per wave and Tank Horse gets +15% health per wave, achieved via `stat_multiplier` export per variant type. This detail is not reflected in the code snippet — it stops at the 50/50 base/variant selection. This is acceptable as a future enhancement; the base wave scaling is already implemented and works.

## Risks / Blockers

### Risk 1: Collision shape not scaled for variant sizes
Tank Horse (scale=1.3) is 30% larger visually but CollisionShape2D radius stays at 20.0 (from horse_base.tscn). This means the hurtbox/hitbox is smaller relative to the sprite. For a top-down arena game this is unlikely to cause gameplay issues, but it is a known asymmetry. Not a blocker.

### Risk 2: Per-variant wave scaling is not in the code snippet
Lines 61–63 of the plan describe a stat multiplier mechanism that is not present in the `_spawn_wave()` code snippet in lines 83–102. The base wave scaling still works (enemy.health, enemy.speed, enemy.contact_damage are overridden per wave), but variant-specific modifiers (+10% speed for Fast, +15% health for Tank) are not yet implemented. Acceptable as a follow-on enhancement, not a blocker for initial implementation.

### Risk 3: AnimatedSprite2D rotation inherited from base scene
`horse_base.tscn` sets `rotation = -1.5708` (90°) on AnimatedSprite2D. Both variant scenes inherit this. Since the LPC horse sprites are side-view, this rotation makes them face the correct direction. No issue identified.

### No blockers identified.
