# Review Artifact — REMED-024

## Finding: EXEC-GODOT-012

**Title:** Player-facing Godot UI reads singleton gameplay state that no runtime code ever updates

**Review date:** 2026-04-17

---

## Finding Detail

`game_over.gd` reads two fields from the `GameManager` autoload singleton that are **never written by any runtime code**:

| Field | Read in | Written by runtime? | Confirmed |
|-------|---------|--------------------|-----------|
| `GameManager.score` | `game_over.gd:10` | **NO** — `add_score()` never called | ✓ |
| `GameManager.max_wave` | `game_over.gd:12` | **NO** — `set_wave()` never called | ✓ |

**Verification against current source:**
- `game_manager.gd` lines 11-17: `add_score()` and `set_wave()` exist as methods but are **never invoked** by any caller in the repo.
- `wave_spawner.gd` `start_wave()` (lines 36-42): increments `_current_wave` internally but never calls `GameManager.set_wave()`.
- `horse_base.gd` `_die()` (lines 86-114): emits `died(contact_damage)` but never calls `GameManager.add_score()`.

---

## Fix Approach Verification

### Fix 1: WaveSpawner → GameManager.set_wave()

**File:** `scripts/wave_spawner.gd`
**Proposed change:** Insert `GameManager.set_wave(_current_wave)` in `start_wave()` after `_current_wave += 1` and before `wave_started.emit(_current_wave)`.

**Soundness:** ✓ Correct. `GameManager.set_wave()` (game_manager.gd lines 14-17) updates both `wave` and `max_wave`, so inserting the call in `start_wave()` before the signal ensures the singleton is synced before any UI reads it.

### Fix 2: HorseBase → GameManager.add_score()

**File:** `scenes/enemies/horse_base.gd`
**Proposed change:** Insert `GameManager.add_score(contact_damage)` in `_die()` before `died.emit(contact_damage)`.

**Soundness:** ✓ Correct. The `contact_damage` value passed to `died.emit()` is the intended score contribution. Calling `GameManager.add_score()` before the signal ensures HUD's local `_score` update (via `hud.gd._on_enemy_died`) and the singleton update both see the same value.
