### Summary

The plan is structurally comprehensive and covers all 5 acceptance criteria. Signal wiring, defensive null-checks, and sourced-asset usage are all documented. However, there is a **material contradiction** between two sections of the plan regarding the HUD scene's root node type — one section shows `CanvasLayer` as root, while another section explicitly recommends `Control` as root to avoid nested `CanvasLayer` stacking. This ambiguity must be resolved before implementation begins.

## Specific Issues

### Issue 1: Root node type contradiction (Material Ambiguity)

**Severity: BLOCKER**

Section 2 (Scene Structure, line 17) shows:
```
hud.tscn (root: CanvasLayer)
```

But Section 5 (HUD Instantiation in main.tscn, lines 151–155) explicitly states:
> "The recommended pattern is: `main.tscn` `CanvasLayer` → children = `[HUD scene root]`  
> **HUD scene root = `Control` (not another `CanvasLayer`)** so it inherits the parent's draw layer."

And line 155:
> **Recommended HUD scene root type: `Control`** — placed inside `CanvasLayer` in main.tscn.

This is a direct contradiction. If `hud.tscn` has `CanvasLayer` as its root, and we instance it as a child of `main.tscn`'s existing `CanvasLayer`, we create nested `CanvasLayer` nodes — which is valid Godot but is not the recommended pattern the plan itself identifies. The plan cannot simultaneously recommend `Control` as the root and show `CanvasLayer` as the root in the scene structure.

**Impact:** Implementation could choose either root type, producing a different scene hierarchy than intended. If `CanvasLayer` root is chosen, the plan's own recommendation to avoid nested `CanvasLayer` is violated. If `Control` root is chosen, the scene structure diagram in Section 2 is incorrect.

**Resolution required:** Plan must pick one and be consistent. The recommended root type should be `Control` (per Section 5's own guidance), and Section 2's scene structure diagram must be updated to reflect `hud.tscn (root: Control)` or the scene diagram must be removed if it cannot be made accurate.

---

### Issue 2: Score signal naming inconsistency

**Severity: LOW (Informational)**

Section 3c (lines 104–119) refers to `score_changed` signal in the narrative but the implementation code uses `died(score_value: int)` from the enemy. The acceptance criteria do not require a specific signal name, but for clarity the plan should align terminology:
- The "score" signal the HUD listens for is the `died(score_value: int)` emitted by `horse_base.gd`.
- The plan correctly maps this in the code block, but the prose calls it "score_changed" which may confuse implementers.

No blocking impact; this is a documentation clarity note.

## Recommended Fix (to resolve REVISE)

1. **Update Section 2** scene structure line to show `hud.tscn (root: Control)` instead of `(root: CanvasLayer)`.
2. **Update Section 5** prose to remove the nested-CanvasLayer alternative ("The HUD scene (rooted at `CanvasLayer`) should replace that bare `CanvasLayer`") — it conflicts with the `Control` root recommendation and adds confusion.
3. **Clarify Section 3c prose** to align signal naming: the HUD subscribes to the `died(score_value: int)` signal from enemies (CORE-002/horse_base.gd), not a separate `score_changed` signal.

After these three corrections, the plan is ready for APPROVAL.
