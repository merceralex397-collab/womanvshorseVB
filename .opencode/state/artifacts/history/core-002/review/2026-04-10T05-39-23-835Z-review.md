# Review: CORE-002 Plan Review

## Verdict
APPROVED

## Reviewer Notes

The plan for CORE-002 is well-structured and covers all acceptance criteria. The technical approach is sound for Godot 4.6, using correct APIs (`move_and_slide()`, `@export`, signal patterns, `class_name`). The sprite orientation workaround for the LPC side-view horses is thoughtfully documented with multiple options. The plan also includes good defensive patterns (alive flag, null checks, monitorable flag reminder).

**Minor concern (non-blocking):** The player reference lookup uses `get_tree().get_first_node_in_group("main")` which depends on a "main" group being set up. The main.tscn root node is named "Main" (capital M), not "main" (lowercase), and is not confirmed to be in a group. If this lookup fails, the horse silently does nothing (returns early in `_physics_process`). A more robust approach like `get_node("/root/Main/Player")` or confirming the group setup would be preferable, but this is a minor issue that the implementer can address during coding.

**Note on `contact_damage`:** The plan correctly defers wiring `contact_damage` to CORE-005 (collision/damage system). The exported variable exists and can be used later.

## Technical Soundness

**Godot 4.6 Patterns Used:**
- `class_name HorseBase extends CharacterBody2D` — correct Godot 4 type registration
- `_physics_process(delta: float)` — correct override signature
- `velocity = direction * speed` then `move_and_slide()` — correct CharacterBody2D pattern
- `@export var health: int = 50` — correct exported variable syntax
- `signal died(score_value: int)` — correct signal declaration
- `area.name == "HitboxArea"` and `area.get_parent().get("attack_damage")` — correct area detection pattern
- `$Sprite.flip_h` — correct AnimatedSprite2D API

**Sprite Orientation:**
Section 3 addresses the LPC Horses side-view issue with multiple workarounds (flip_h, rotate AnimatedSprite2D node -90°). The plan acknowledges this needs visual verification.

**Collision Layers:**
Section 1 mentions "Solids" for physical bodies and "Tools" for hurtbox area — the implementer should verify these match the actual project collision layer setup.

**Defensive Patterns:**
- `_alive` guard flag prevents double-kill
- Null checks on player reference
- `if not _alive: return` guards in `take_damage()`

## Issues (none requiring revision)

The plan is approvable as-is. The implementer should be aware of the following during coding:

1. **Player reference robustness:** Consider using `get_node("/root/Main/Player")` if the "main" group approach fails during implementation testing.

2. **Collision layers:** Verify that the "Solids" and "Tools" layer names match what's defined in `project.godot` before setting `collision_layer` and `collision_mask`.

3. **HurtboxArea `monitorable`:** The plan documents this (Section 5), but the implementer must ensure `monitorable = true` is set on the HurtboxArea Area2D node in the scene.

4. **Score value:** The `died.emit(10)` hardcodes score=10. An exported `score_value` variable would be more flexible for wave spawner tuning (the plan notes this in Section 2).
